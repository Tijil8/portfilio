system 01 · the marketplace

Three roles, one platform

The product is a two-sided marketplace with an operations layer on top. Every role sees a different surface, backed by the same data model with strict row-level access.

◆ Employer

Posts jobs, manages listings, and reviews incoming applications with notifications on each one.

◆ Candidate

Builds a profile, browses roles, and applies — with application state tracked end-to-end.

◆ Admin

A control panel over jobs, reviews, case studies and users — the operations layer that runs the platform.

system 02 · auth & access

Auth, sessions, and row-level security

A full authentication stack — signup, email verification, login, forgot/reset password — with hashed credentials and signed JWT sessions. The real access control lives in the database: Postgres row-level security policies decide, per row, what an employer, candidate, or admin can read and write. 

-- access is enforced in the database, not just the app
create policy "employers manage their own jobs"
  on jobs for all
  using ( auth.uid() = employer_id );

create policy "candidates read published jobs"
  on jobs for select
  using ( status = 'published' );
  • Password hashing with bcrypt; stateless sessions via signed JWTs
  • Email verification + password reset flows, transactional email on send
  • Google OAuth wired in alongside email/password
system 03 · the api

A typed API behind every action

Next.js route handlers back the whole product — auth, jobs, applications, reviews, notifications and newsletter — all in TypeScript against a Supabase Postgres backend.

candidate applies→ /api/jobs/[id]→ RLS-checked write→ notify employer→ email out
  • Job CRUD with per-listing endpoints and application tracking
  • Notification pipeline that emails the employer on every new application
  • Reviews & case-studies managed through the admin API
  • Newsletter subscription with a transactional-email backend
